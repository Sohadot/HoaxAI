# Hoax.ai Master Execution Plan

## Purpose

This plan defines the strict phase order, gates, and stop conditions governing Hoax.ai development.

Hoax.ai is a Category Intelligence Factory. Factory layers must be built in sequence. Skipping phases creates trust debt that cannot be recovered through language.

## Phase Order

The following sequence is mandatory:

| Order | Phase Principle |
|-------|-----------------|
| 1 | **Thesis before interface** |
| 2 | **Governance before tools** |
| 3 | **Self-application before external claims** |
| 4 | **Integrity standard before reference expansion** |
| 5 | **Ontology before scale** |
| 6 | **Standard before scoring** |
| 7 | **Protocol before classifier** |
| 8 | **Source discipline before reference pages** |
| 9 | **Reference layer before monetization** |
| 10 | **Trust before revenue** |
| 11 | **GitHub-first before DNS or Cloudflare** |

No sprint may begin if its prerequisite phases are incomplete and validated.

## Phase Gates

A phase gate is a validation checkpoint. The next phase does not open until the current phase passes its gate.

| Gate | Prerequisite | Validation |
|------|--------------|------------|
| G0 | Foundation governance | Sprint 0A audit passed |
| G1 | Public thesis surface | Sprint 1 audit passed |
| G1A | Artifact–subject separation | Sprint 1A audit passed |
| G1B | Sovereign integrity foundation | Sprint 1B audit passed |
| G2 | Evidence posture taxonomy | Taxonomy file validated, governance aligned |
| G3 | Classification protocol | Protocol documented, output boundaries defined |
| G4 | Classifier v0 | Tool deployed with no-verdict disclaimer, no upload |
| G5 | Reference layer v1 | Pages reference-grade, source-disciplined, no thin pages |
| G6 | Source registry | Claim mapping and citation policy active |
| G7 | Interface embodiment | Visual system aligned with interface thesis |
| G8 | GitHub public completion | All links working, no false claims, mobile-safe |
| G9 | DNS and Cloudflare | Public foundation complete and validated |

**Gate G1B is active.** Ontology, standard, protocol, classifier, and reference pages cannot proceed until Sprint 1B passes.

## Sprint Closure Rules

A sprint closes only when:

1. All deliverables listed in ROADMAP.md for that sprint are complete.
2. A sprint audit file is created documenting files reviewed, files changed, and validation checklist.
3. No prohibited items from the sprint scope were created.
4. Governance consistency is confirmed across affected files.
5. DECISION_LOG.md is updated if the sprint introduces new decisions.
6. Artifact–Subject Separation Doctrine remains intact.

A sprint marked COMPLETE without an audit file is not closed.

## Stop Conditions

Development must stop and remediate when:

- a public claim violates CLAIM_POLICY.md or GOVERNANCE_BOUNDARY.md;
- a future capability is described as existing;
- an unsupported external factual claim enters the public surface or evidence ledger;
- artifact risk language implies subject accusation;
- a tool is deployed without output boundary and protocol definition;
- a reference page is published without source discipline;
- a route is created without defined reference purpose;
- monetization is introduced before reference layer validation;
- DNS or Cloudflare work begins before GitHub public foundation is complete;
- an absolute security, detection, or truth verdict claim appears in any file.

Stop conditions trigger remediation before the next sprint opens.

## Quality Gates

Every deliverable must pass applicable quality gates before sprint closure:

| Gate | Criterion |
|------|-----------|
| Claim gate | Claims categorized, ledger-recorded, governance-aligned |
| Page gate | Purpose defined, reference value present, no thin content |
| Route gate | Purpose mapped, no placeholders, links resolve |
| Source gate | External facts cited and classified |
| Tool gate | Output boundary defined, no verdict language, no subject accusation |
| Interface gate | Evidence structure embodied, no false-authority design |
| Security gate | No known unresolved critical issues after validation |

## Structural Rules

### No Route Without Purpose

Every route must map to a defined sprint deliverable or reference function.

Routes created for SEO volume, placeholder navigation, or undifferentiated content are prohibited.

### No Page Without Reference Value

Every public page must contribute thesis, governance visibility, taxonomy, standard, protocol, or source-disciplined reference content.

Thin pages, keyword-stuffed pages, and undifferentiated landing pages are prohibited.

### No Tool Without Output Boundary

Every tool must have defined output language, confidence framing, no-verdict disclaimer, and artifact–subject separation before deployment.

Classifier, API, and workflow tools require protocol and standard completion first.
