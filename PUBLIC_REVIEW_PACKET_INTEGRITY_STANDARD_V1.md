# Public Review Packet Integrity Standard v1

Sprint 103 — Public Reference Review Packet Integrity Audit v1  
**Decision:** DEC-121

## Review Packet Integrity Standard Statement

The reviewer packet must organize public inspection without creating conclusions, reports, private access, transactions, or operational claims. Integrity audits validate production pages — not abstract governance alone.

## Reviewer packet route-count requirements

- Sitemap must remain exactly 68 URLs during integrity audit sprints unless an expansion gate authorizes change
- Route registry must remain exactly 68 entries
- public-file-registry must stay aligned if used
- No stale route-count language in public HTML conflicting with current counts

## Reviewer packet component preservation requirements

Every reviewer-packet page must preserve Reference summary, Packet purpose, Review path, What this page supports, What this page does not claim, Reference Answer, Source Confidence, Cite This Reference, Retrieval Capsule, Boundary reminder, Non-transactional review boundary, and page-end reference navigation.

## Reviewer packet metadata requirements

Exactly one H1, canonical URL, title, meta description, Open Graph title, and Open Graph description on every public HTML page.

## Reviewer packet link requirements

- Link to `/reviewer-packet/`
- At least 3 strategic or external-review routes
- At least 5 reference, utility, or pathway routes
- At least 2 sibling reviewer-packet routes
- No broken internal route links

## Reviewer packet role clarity requirements

Each page must state the packet is a public reference packet and review organizer — not a report, private data room, transaction surface, scorecard, verdict system, or legal or financial representation.

## Non-verdict requirements

No detector claims, verdict language as capability, scoring, or automated report promises.

## Non-transactional requirements

No pricing statements, transaction pages, acquisition term documents, representative mandates, or legal or financial representation.

## Private data-room prohibition

No private data-room access claims in public HTML.

## Downloadable report prohibition

No downloadable report behavior or promises in public HTML.

## Forbidden regression patterns

Forbidden exact n-grams in public HTML include detector, upload, scoring, verdict, transaction, pricing, private data room, and downloadable report phrasing listed in Sprint 103 specification. Safe negative language is allowed.

## Future review-packet audit rules

- Reviewer Packet Integrity Snapshot on `/reviewer-packet/` must reflect current route counts
- Integrity audits may repair metadata, links, components, and boundary wording without adding routes
- Historical governance documents are not rewritten for current public copy rules
