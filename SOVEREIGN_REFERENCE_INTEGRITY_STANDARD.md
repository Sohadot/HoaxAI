# Hoax.ai Sovereign Reference Integrity Standard

## Purpose

This standard defines what Hoax.ai must satisfy before it can call itself reference-grade.

Reference authority must be earned by structure, not claimed by language.

A site can claim trust. A governed asset must make trust inspectable.

## Reference Authority Rule

Hoax.ai may not claim reference authority through marketing language, domain prestige, or assertion.

Reference authority is earned when structural integrity requirements across claims, pages, routes, sources, tools, and governance are satisfied and validated.

Until validation passes, Hoax.ai remains a governed asset under development — not a certified reference authority.

## Claim Integrity

- All public claims must align with CLAIM_POLICY.md and GOVERNANCE_BOUNDARY.md.
- Foundational claims must be recorded in the evidence ledger with appropriate evidence posture.
- Conceptual thesis must be separated from operational and factual claims.
- Future capabilities must use `planned_not_claimed` posture and must not appear as existing.
- External factual claims must be source-supported before publication.
- Artifact–subject separation must be preserved in all claim language.

## Page Integrity

- Every public page must have a defined purpose aligned with category infrastructure.
- No thin pages. No keyword-stuffed content. No placeholder reference pages.
- Every page must contribute reference value: thesis, governance visibility, taxonomy, standard, protocol, or source-disciplined explanation.
- Pages must not imply tool capability, detection authority, or truth verdicts beyond governance boundaries.

## Route Integrity

- No route without purpose.
- Every route must map to a defined sprint deliverable or reference function.
- No placeholder routes. No broken routes. No routes created for SEO volume alone.
- Route expansion follows MASTER_EXECUTION_PLAN.md phase order.

## Link Integrity

- Internal links must resolve and serve defined reference or governance functions.
- No broken internal links on the public surface.
- Cross-references between governance files, ledger entries, and public pages must remain consistent.
- Link structure must support inspectability, not decorative navigation.

## Source Integrity

- External factual claims require source support per SOURCE_POLICY.md.
- Reference pages require cited, classified sources before publication.
- Source registry discipline applies once Sprint 6 source registry is established.
- Unsupported statistics, examples, or assertions must not appear on reference pages.

## Technical Integrity

- Public surface must be static, accessible, mobile-stable, and performance-safe.
- No unresolved critical technical defects after validation.
- HTML structure must support accessibility and SEO discipline without thin-content expansion.
- No JavaScript dependencies or external libraries unless explicitly approved by decision log entry.

## Security Posture

Hoax.ai must maintain a validated security posture without claiming impossible guarantees.

Approved wording:

> No known unresolved critical security, technical, accessibility, or performance issues after validation.

Prohibited wording:

- No security gaps.
- No technical gaps.
- Perfectly secure.

Security posture is validated, not asserted. Validation records belong in sprint audit files.

## SEO Discipline

- SEO serves reference infrastructure, not traffic harvesting.
- No thin pages. No fake reference pages. No keyword stuffing.
- Sitemap and robots files must reflect actual public routes only.
- Canonical URLs must align with governance-approved domain connection timing.
- SEO expansion follows reference layer completion, not precedes it.

## Interface Integrity

- Interface must embody evidence structure per INTERFACE_THESIS.md and DEC-005.
- No fake scanner aesthetics. No upload tool presentation. No truth meters or risk dashboards.
- No cyber-gaming threat visuals. No sensational red-alert accusation design.
- Interface language must align with artifact–subject separation and evidence posture framing.

## Tool Integrity

- No tool without output boundary.
- Classifier, API, or workflow tools require protocol, standard, and governance boundary before deployment.
- Tools must not imply absolute detection, truth verdicts, or subject accusation.
- Tool outputs must communicate bounded evidence posture only.

## Monetization Integrity

- Monetization must preserve trust per MONETIZATION_BOUNDARY.md and DEC-007.
- No monetization before reference layer and trust foundation are established.
- Revenue products must not convert evidence risk into subject accusation or sensational claims.
- Trust before revenue.

## Acquisition Integrity

- Acquisition positioning must reflect category infrastructure value, not domain speculation alone.
- Buyer logic per BUYER_LOGIC.md must be supported by inspectable governance and reference assets.
- Strategic value derives from governed intelligence layers, not marketing assertions.

## Validation Rule

Hoax.ai may claim reference-grade status only when:

1. All integrity dimensions above are satisfied for the current sprint scope.
2. A sprint audit file confirms validation with explicit pass criteria.
3. No known unresolved critical security, technical, accessibility, or performance issues remain after validation.
4. Evidence ledger, governance files, and public surface are consistent.
5. Artifact–Subject Separation Doctrine remains intact.

Validation is recorded in sprint audit files. It is not self-certified by language.

## Governing Sentences

**Reference authority must be earned by structure, not claimed by language.**

**A site can claim trust. A governed asset must make trust inspectable.**
