## Purpose

This document prepares a future **public interface thesis** and **evidence-field design foundation** from the existing public protocol at `/protocol/evidence-posture/`. It is not a public page, not an interface route, and not operational governance expansion. **No public interface route is created in Sprint 62.**

---

## Evidence Field, not Detector Dashboard

A future Hoax.ai interface must read as an **evidence field**—a bounded space where artifact condition, claim scope, source basis, limitations, and posture states are reviewed in sequence—not as a detector dashboard with fake/real buttons, confidence meters, or verdict cards.

Detector-dashboard framing is prohibited because it:

- Implies automated classification where only review sequence exists.
- Converts posture states into verdict states visually.
- Invites upload-and-score behavior before governance authorizes it.
- Undermines artifact-subject separation by foregrounding subject implication.

Interface thesis should embody **evidence posture**—what the artifact and claim support under boundary rules—not decorative AI aesthetics or scanner metaphors.

---

## Protocol Path, not Scoring Funnel

Evidence Posture Protocol v1 Draft defines EP-P01 through EP-P17 as an ordered review path. A future interface may visualize this path as panels or stages aligned to protocol steps—not as a scoring funnel, rank ladder, or percentage-confidence workflow.

| Protocol step | Potential interface panel |
|---------------|---------------------------|
| EP-P01–EP-P02 | Artifact and claim identification |
| EP-P03–EP-P07 | Source basis and traceability |
| EP-P05–EP-P06, EP-P10 | Provenance and context review |
| EP-P08–EP-P11 | Chain, fragility, and drift review |
| EP-P12–EP-P14 | Limitation, interpretation risk, attribution boundary |
| EP-P15–EP-P17 | Output boundary, posture assignment, bounded language |

No panel may imply numeric scoring, classifier output, or automated verdict generation.

---

## Boundary States, not Verdict States

Posture states from the standard—Supported, Qualified, Limited, Not Assessable, Out of Scope—must remain visually distinct from truth verdicts, guilt labels, fake/real badges, or moderation outcomes.

Boundary states that must remain visually clear:

- **Not Assessable** — responsible limit, not hidden false judgment
- **Limited** — narrow envelope, not weakness accusation
- **Attribution boundary** — artifact scope, not subject responsibility
- **Output boundary** — language refusal, not censorship verdict
- **Prohibited inference** — what the interface must not display

---

## Review Sequence, not Automated Classification

The protocol defines **review sequence**. The standard defines **allowed and prohibited language**. The reference layer defines **concepts**. A future interface may help a reviewer walk EP-P01 through EP-P17—it must not automate classification, issue fake/real results, or replace human institutional judgment with a public verdict button.

Human-readable protocol visualization is authorized in principle. Public engine behavior is not.

---

## Output Restraint, not Result Generation

Interface design must foreground **output restraint**: limitation statements, provenance caveats, context caveats, attribution boundary caveats, and interpretation risk caveats. It must not foreground result generation, headline certainty, or compressed model-summary overreach.

EP-P17 (Produce bounded language only) is the terminal discipline. Any interface that suggests results before boundary review is out of alignment with the protocol.

---

## Why No Public Interface in Sprint 62

Sprint 62 ends at **interface readiness**. The public protocol exists; the standard authorizes language; the reference layer supplies concepts. Creating a public interface route now would:

- Imply tool behavior before interface thesis is governed.
- Risk detector-dashboard conventions by default.
- Bypass evidence-field design discipline.
- Expand operational surface without separate gates.

No `/interface/` route, no public HTML interface page, and no public tool behavior is authorized.

---

## Future Interface Boundary

A future interface may:

- Visualize protocol steps as navigable review panels.
- Display posture states with standard-aligned labels.
- Show boundary refusals and prohibited inference warnings.
- Link to reference concepts and standard principles at each step.
- Remain static, reference-grade, and non-operational until separately authorized.

A future interface must not:

- Create upload workflows, scoring systems, or classifier surfaces.
- Expose API endpoints, analytics dashboards, or forms on the public surface.
- Use JavaScript-driven verdict behavior on the public surface.
- Look like an AI detector, scanner, or fake/real tool.
- Authorize language prohibited by Evidence Posture Standard v1.

The protocol cannot exceed the standard. The interface cannot exceed the protocol and standard together.
