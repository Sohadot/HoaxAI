# Internal Non-Public Engine Prototype Charter

## 1. Charter Statement

**A future Hoax.ai internal prototype may only test whether Evidence Posture Standard v1, Evidence Posture Protocol v1 Draft, Engine Boundary Charter, Evidence Posture Engine Model v0, and Output Language Guardrail Model v1 can be composed into a non-public, non-executable review architecture without producing verdicts, scores, classifications, accusations, public outputs, or operational decisions.**

This charter defines prototype **admissibility**. It does not authorize prototype construction or execution.

## 2. Prototype Status

Sprint 70 establishes governance only:

- **no prototype exists**
- **no prototype is authorized by Sprint 70**
- **no public engine exists**
- **no private execution engine exists**
- **no input system exists**
- **no output generator exists**
- **no UI exists**
- **no API exists**
- **no scoring system exists**

**Sprint 70 does not authorize a prototype.** Sprint 71 may only consider a Controlled Internal Prototype v0 after separate explicit authorization.

## 3. Source Authority

Any future internal prototype must obey:

- Evidence Posture Standard v1
- Evidence Posture Protocol v1 Draft
- Engine Boundary Charter
- Evidence Posture Engine Model v0
- Output Language Guardrail Model v1
- Output Boundary
- Evidence Limitation
- Attribution Boundary
- Interpretation Risk
- Artifact–Subject Separation
- Claim–Source Traceability
- Public Reference SEO Authority Map v1

## 4. Prototype Purpose

A future prototype may only test:

- protocol-step sequencing
- posture-state mapping
- boundary-check activation
- caveat-trigger mapping
- output-language guardrail application
- schema compatibility
- admissibility checks
- failure-mode detection
- non-verdict behavior

A future prototype must **not** test:

- fake/real detection
- identity attribution
- intent inference
- subject guilt
- fraud finding
- manipulation proof
- public moderation
- legal judgment
- scoring
- confidence percentages
- upload classification
- external fact verification
- public reporting

## 5. Prototype Admissibility Conditions

A future prototype is admissible only if:

- it is non-public
- it has no route
- it has no sitemap entry
- it has no public link
- it has no upload interface
- it has no form
- it has no API endpoint
- it has no JavaScript public behavior
- it uses only governed fixtures
- it outputs only bounded, non-verdict structures
- it preserves artifact-subject separation
- it runs only after separate sprint authorization

See `INTERNAL_PROTOTYPE_ADMISSIBILITY_MODEL.md` for the full admissibility model.

## 6. Environment Boundary

Permitted future environments:

- local-only prototype environment
- non-public test directory under governed internal paths
- no public route
- no deployment
- no analytics
- no external API calls
- no public asset pipeline
- no crawler exposure
- no sitemap exposure

## 7. Fixture Policy

Future prototype fixtures must be synthetic, neutral, case-neutral, non-personal, non-accusatory, and free of private personal data. See `INTERNAL_PROTOTYPE_FIXTURE_POLICY.md`.

**Forbidden fixture types:** real person accusation, active news event, political claim, celebrity claim, company fraud claim, medical misinformation case, legal dispute, public scandal, private screenshots, personal messages, uploaded user files, copyrighted article reproduction, external fact-check target.

## 8. Input Boundary

**No input system is created in Sprint 70.**

Future prototype input, if ever authorized, must be fixture-bound and must not permit user upload, public input, URL submission, live web lookup, external API ingestion, account data ingestion, private data ingestion, image/video upload, or real-world accusation submission.

## 9. Output Boundary

Future prototype output may only be structured as:

- posture_state_candidate
- active_boundary_checks
- triggered_caveats
- prohibited_language_blocks
- required_output_constraints
- not_assessable_reason
- out_of_scope_reason
- guardrail_failure_flags

Future prototype output must **not** include:

- fake/real result
- truth/falsity result
- score
- confidence percentage
- subject guilt
- deception finding
- manipulation proof
- fraud accusation
- legal conclusion
- moderation action
- public result card
- downloadable report
- shareable claim result

## 10. Prototype Execution Boundary

**Sprint 70 authorizes no execution.**

A later sprint may only authorize a controlled internal prototype if:

- this charter passes validation
- the fixture policy passes validation
- the guardrail model passes validation
- the engine model passes validation
- a prototype-specific validator is written before implementation
- the prototype cannot be publicly accessed
- the prototype cannot emit prohibited output

## 11. Prototype File Boundary

**Future allowed file categories:**

- internal model adapter
- fixture parser
- validation harness
- output guardrail checker
- no-public-route test harness
- local-only documentation

**Prohibited file categories:**

- public route
- client-side UI
- upload page
- API endpoint
- public JavaScript
- score renderer
- detector result component
- share/export report component
- crawler-exposed demo

Existing files at `_internal_prototypes/evidence-posture-workbench/` remain governed and must not be modified without explicit sprint authorization.

## 12. Failure Modes

- public-route leakage
- sitemap leakage
- prototype-as-product drift
- upload implication
- detector drift
- output-generator drift
- score drift
- verdict leakage
- subject-transfer leakage
- external fact-check drift
- live-event fixture drift
- case-specific accusation drift
- SEO bait prototype drift

Sprint 70 does not authorize a prototype. It only defines the conditions under which **Sprint 71 — Controlled Internal Prototype v0 Authorization Package** may consider a controlled internal prototype after separate explicit authorization and validation pass.
