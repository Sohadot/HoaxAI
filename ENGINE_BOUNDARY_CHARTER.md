# Engine Boundary Charter

## Purpose

Hoax.ai must protect its category identity before any engine modeling begins. This charter defines what any future engine may and may not do within the evidence posture asset. It is an internal asset-quality boundary, not an authorization to build or deploy an engine.

## Current State

The public surface remains exactly nineteen URLs across reference, standard, protocol, and static interface layers. Hoax.ai has no public engine, no classifier, no upload workflow, no scoring system, no API, no analytics, no forms, no JavaScript tool behavior, and no automated verdict interface.

## Engine Drift Definition

Engine drift occurs when Hoax.ai becomes a generic detector, scanner, upload tool, scoring product, or fake/real verdict system instead of a governed evidence posture reference asset. Engine drift includes:

- upload-centered workflows
- detector dashboard framing
- confidence percentage or numeric score outputs
- fake/real or authenticity certification language
- subject accusation or deception proof claims
- public API endpoints without explicit governed authorization
- automated classification presented as operational product behavior
- monetization surfaces tied to detection or scoring

## Permitted Future Engine Modeling Scope (Not Authorized Now)

This charter does not authorize engine construction. If a future governed Engine Model v0 sprint is explicitly authorized, modeling may consider:

- internal evidence posture reasoning boundaries
- artifact-subject separation in any future output model
- output boundary constraints aligned with Evidence Posture Standard v1
- refusal and not-assessable posture states
- provenance and source discipline without truth verdicts
- governance gates before any public engine exposure

## Prohibited Engine Behaviors (Always)

No future engine work may introduce, imply, or prepare for:

- public upload interfaces
- public scoring or ranking systems
- public classifier dashboards
- public fake/real verdict cards
- subject accusation outputs
- ungoverned API endpoints
- analytics on public pages
- JavaScript-driven operational tool behavior on the public surface
- bypass of validate_all.py or publisher quality gates
- expansion beyond registered routes and sitemap governance

## Preconditions Before Engine Model v0 May Be Considered

1. Engine Boundary Charter exists and passes validation.
2. Public Reference SEO Authority Map v1 exists and passes validation.
3. Sprint 67 validation passes with validate_all.py PASS.
4. Sitemap remains exactly nineteen URLs with no unauthorized route expansion.
5. A separate explicit governed decision authorizes Engine Model v0 — not this charter.

## Relationship to Reference, Standard, Protocol, and Interface Layers

Any future engine must remain subordinate to:

- the twelve public reference pages and category language layer
- Evidence Posture Standard v1
- Evidence Posture Protocol v1 Draft
- Evidence Field Interface Thesis with static non-operational embodiment

The engine may not redefine category language, override output boundaries, or convert the interface thesis into an operational product surface without separate governed authorization.

## Governance and Validation

This charter is enforced through repository validators and publisher quality gates. Changes to engine boundaries require governed decisions, claim traceability where applicable, and validate_all.py PASS before any downstream sprint may proceed.
