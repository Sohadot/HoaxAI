# Hoax.ai Structured Data and Semantic SEO Governance

**Version:** v1.0.0  
**Status:** governed_internal_semantic_seo_standard  
**Maturity:** pre_reference_publication_seo_governance  
**Decision:** DEC-033

## A. Purpose

This document governs semantic SEO, metadata, structured data, canonical signals, and future reference page SEO for Hoax.ai.

## B. Non-Purpose

This governance does **not**:

- create public pages;
- create draft pages;
- create routes;
- create sitemap expansion;
- create deployment readiness;
- create DNS or Cloudflare work;
- create public classifier functionality;
- create a public tool;
- create upload functionality;
- create forms;
- create analytics;
- create monetization;
- create external factual claims;
- authorize publisher output.

## C. Governing Principle

**SEO must describe governed meaning, not manufacture authority.**

**Structured data must not claim product, tool, detector, service, or software capability before those gates exist.**

## D. Semantic SEO Definition

Semantic SEO is the disciplined use of titles, descriptions, headings, canonical URLs, structured data, internal links, and future page metadata to clarify Hoax.ai's governed reference concepts without keyword stuffing, route inflation, false capability claims, unsupported authority, or detector/tool implication.

## E. SEO Is Not

SEO is not:

- keyword stuffing;
- search-volume-first publishing;
- route inflation;
- content volume strategy;
- detector keyword capture;
- fake/real keyword capture;
- clickbait metadata;
- unsupported superiority;
- schema overclaiming;
- hiding weak content behind strong metadata.

## F. Allowed Current Metadata Posture

Current homepage metadata may express:

- Hoax.ai as a governed evidence posture reference asset;
- Hoax.ai as not a truth machine;
- evidence structure and trust posture;
- governance boundaries;
- category thesis;
- reference infrastructure under governance.

Current homepage must not imply live classifier, active detector, upload, scan/check, scoring, API, commercial service, software application, product availability, deployment completion, truth certification, forensic validity, or fraud/guilt determination.

## G. Structured Data Rules

**Allowed currently:** WebSite and WebPage if bounded and non-tool-implying; Organization only if bounded identity without service/tool provider implication; BreadcrumbList only when multiple public routes exist and route governance permits.

**Prohibited currently:** SoftwareApplication; Product; Service; APIReference; HowTo; FAQPage for claim-heavy SEO capture; Review; AggregateRating; Dataset unless future source governance approves; medical/legal/financial/forensic schemas; schemas implying active detector, classifier, upload, scan, scoring, or commercial tool.

## H. Title and Meta Description Rules

Future titles and descriptions must describe governed concept, match page substance, avoid active tool language, detector/fake/real/scan/upload/scoring promises, unsupported superiority, clickbait, and unmapped external factual claims.

## I. Heading Rules

H1 and headings must reflect the governed concept, not chase keywords independently, not imply live tool capability, not use fake/real detector framing, and not turn artifact posture into subject accusation.

## J. Canonical and Route SEO Rules

Canonical and route SEO must align with route registry and sitemap eligibility, not canonicalize planned routes, not create indexable placeholders, not imply deployment completion, and not support programmatic route inflation.

## K. Internal Semantic Link Rules

Future semantic internal links must connect governed concepts, use descriptive anchor text, point only to registered eligible routes, avoid planned route links, and support reference comprehension—not link stuffing.

## L. Publisher SEO Rule

The publisher may prepare SEO metadata only after candidate registration, page family assignment, reference thesis definition, claim scope definition, source scope definition, content quality gate satisfaction, and route purpose definition. Publisher SEO must not create claims unsupported by body, evidence ledger, claim-source map, and source registry.
