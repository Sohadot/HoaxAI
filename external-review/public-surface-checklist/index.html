Public Surface Review Checklist

Page role: Public Surface Review Checklist

Reference summary: This checklist gives reviewers static sections for inspecting route depth, metadata, utilities, concepts, pathways, citation blocks, and boundaries.

Review purpose

The Public Surface Review Checklist gives external reviewers a static reference checklist for inspecting Hoax.ai's published surface. It is not an interactive form, scoring system, or automated report.

Review path

  1. Confirm route count via release integrity snapshot.
  2. Sample utility usefulness.
  3. Read concept clarity on core units.
  4. Check strategic entry clarity.
  5. Review readiness surface depth.
  6. Return to reviewer map for area orientation.

Checklist sections

Route count and public surface

Confirm the homepage snapshot and sitemap-aligned route groups match expectations for a 104-route public reference system. Cross-check public release and indexation integrity — registry, sitemap, robots, and canonical alignment without marketing-grade metadata drift.

Metadata and canonical clarity

Each route should expose title, meta description, canonical URL, and Open Graph fields suitable for citation.

Utility usefulness

Utilities such as the Manual Evidence Checklist and Evidence Posture Map should teach manual reasoning without operational promises.

Concept definition clarity

Core and deep concepts should define evidence-risk vocabulary with Reference Answer and Source Confidence blocks.

Pathway usability

Pathways should route by evidence condition without case examples or accusation language.

Citation and retrieval components

Pages should include Cite This Reference and Retrieval Capsule sections for agent-safe quoting.

Source-confidence blocks

Support types should state what pages can and cannot support.

Boundary reminders

Non-verdict and non-detector boundaries should appear on strategic and external-review pages.

Strategic readiness

Public Reference Surface Readiness should align with visible utilities and concepts.

Non-transactional framing

Readiness and external-review pages should remain strategic review surfaces, not transaction pages.

Checklist depth

Concept review should confirm that core and deep reference units define evidence-risk vocabulary with stable anchors for human readers and automated readers alike.

Governance-aware review may read standard and protocol pages to understand posture vocabulary, but external-review pages do not themselves constitute legal review or financial representation.

Future external-review expansion may add more inspection guides, but Sprint 101 authorizes only static reference pages with no chatbot behavior, generator behavior, or public API endpoints.

Reviewers documenting findings should cite canonical URLs and boundary reminders rather than paraphrasing Hoax.ai as a detector, verdict engine, or upload tool.

Inspection discipline matters because synthetic-media discourse often collapses evidence conditions into shareable conclusions; Hoax.ai's asset value is slowing that collapse through reference structure.

External review readiness therefore complements release integrity: one confirms the surface is whole, the other teaches how to walk it responsibly.

External review on Hoax.ai means walking the published reference system as a governed knowledge surface rather than treating any page as a hidden product demo or operational endpoint.

Reviewers should compare visible utilities, concept definitions, pathway logic, and strategic layers against the project's stated refusal to issue binary authenticity labels, numeric certainty outputs, or case-specific conclusions.

Human diligence benefits when reviewers start from the homepage release integrity snapshot, then sample entry points, narratives, acquisition-readiness routes, and boundary pages before forming category judgments.

AI-assisted review should retrieve Reference Answer blocks, Source Confidence sections, citation blocks, and retrieval capsules instead of inferring authority from page titles or navigation labels alone.

The external-review layer exists because release integrity alone does not teach reviewers how to sequence inspection across utilities, concepts, pathways, and strategic surfaces.

Partners comparing reference systems should note how external-review pages link outward to evidence-risk vocabulary rather than trapping visitors in a single landing funnel or contact-oriented workflow.

Category thesis review should read narrative pages alongside core and deep reference concepts so evidence-risk language is understood as infrastructure, not as marketing copy detached from route depth.

Pathway review should confirm that condition-based journeys remain manual, non-operational, and free of file provision for analysis surfaces or automated judgment promises.
